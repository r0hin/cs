Changes to my Solution

Introductory

Modifying code is one of the most crucial part of the design system as just looking at the design cycle, there are arrows pointing everywhere meaning you can be at any point and you will be able to go anywhere (ie. redo a process, change your goals, etc). It is almost impossible to start with a plan and follow it perfectly as there is always either shortcomings or you complete more than expected. This solution is no exception and there have been many meaningful changes to both the code, design and plan.

Changes to my Code

The actual code part of my code changed a lot. For example, I was planning on adding images such as profile pictures to the app, but decided not to as there was not enough time to add it and just hooking it up with pyrebase which has limited support took up the majority of the time. Secondly, I opted to use a really small web-based account creation page instead of on tkinter becuase on pyrebase there actually isn't any support for creating and managing email accounts without signing in with Google. Thirdly and finally, I changed the way messages are displayed, instead, I decided on a material design type card style.

Changes to my Plan

The plan which you can view above was also altered slightly but not very much. I followed the plan as close as I could, however, work period 5-6 which consisted of managing data from the database and formatting data took some more troubleshooting and effort than the other parts in the timeline. Secondly, many things which were said to be completed in one work period have to have been postponed to home or even added to the list for another work period, so it was hard to keep on the pace of the work period which is why I had to cut back on some corners in the final product such as not building the application up.

Changes to the Design

The original design plan was meant to be a modern and good looking design but after trying to make it look as good as possible, I concluded that it would be better to just have it a simple app to send quick pointless messages to. Since tkinter is not the most graphic library and not the best framework for creating good looking designs, I figured this one was pretty impossible so I decided to center things, add some shadows and that was pretty much all I could do. So there weren't very many changes to the plan of the design except for maybe one or two very major ones.

Conclusion

In conclusion the use of modification during the design cycle was very important to completing this software and any project for that matter and if I couldn't make those changes, there likely wouldn't have been any product released. To get the basic application out there while making changes to the code, plan and even design part of it was definitely worth it as the alternative of not having an app at all is obviously not as good (bad).

Error Checking and Testing

Introduction

Error checking is another very important element to the success of any solution. In this case, Python is a very easy language to error check and I have took advantage of this to incorporate error checking into my app. In Python, there are three types of errors. Syntax errors, which are the easiest to fix and are mistakes with the writing of the code. Secondly, logic errors which are usually difficult to find and stem from some kind of logically issue within the code. Finally, runtime errors are code errors where python simply doesn't understand what you are trying to say. For example things like dividing by zero, and accessing files that don't exist will produce a runtime error.

Relevance to my Code

In order to ensure non of the above errors would have an effect on my application, I employed some different steps to help me fix these issues. Firstly, to fix the syntax error, I just double cheked my code and made sure everything ran correctly which was very easy. To fix logic errors, I just checked to make sure that everything ran correctly through day-to-day usage and I didn't have to deal with this kind of error after grasping the fact that Python runs sequentially unlike many other languages. Finally, runtime errors just did'nt occur in my code because of good practise and making sure all files exists, try statements, etc.

Solutions

Many issues such as network issues, unforseen database issues are also possible. The way I fixed this was by wrapping common error culprits with a try statement and excepting it with a tkinter create element which esentially puts a message onscreen informing the useer that an error occured and to try restarting the app / computer / internet device.

Improvements and Errors

Introduction

In any given software application and solution, there are always available improvements and new features. For a version two or future update of the software, those features may be added. In relevance to my solution, the same principle definitely applies; there are many features and future improvements which could be made to my program to make it more useful and complete it's tasks better. The following points are: UI / UX, Images / More content, More put-together. I will expand on each one below:

More Content / Profile Pictures / Images

With more time and more resources, it would have been very useful to have an option to upload images, add profile pictures and more to add to the identify of the users and the app in general. Adding profile pictures would allow users to not worry about impersonation while not revealing their personal information. Next, being able to upload images would be helpful such as sharing pictures with people related to what they are talking about. Finally, making it more complete instead of just a text box such as likes and agrees would make it more complete and more dynamic than just a public chatroom type application and less of social media.

UI / UX

Another potential improvement would be the upgrade of the UI. Right now, it is a mix of really old styles and material design cards. Trying to find more components especially buttons and inputs would make it look a whole lot nicer instead of a 2006 windows xp program. Additionally, this would also allow people to feel more comfortable using the app because I for one would not enjoy using an app that looks liike it was from a decade ago as a daily or even a weekly application. Finally, upgrading the UI would improve the overall UX of the application and may even attract more users and overal just improve the experience of existing users and make it more easy on the eyes to those who are used to good looking interfaces instead of traditional ones.

More Put-together

From the title it is hard to understand what I mean by this. Adding things such as a documentation and terms and conditions and a dedicated website or even a forums would just make it look more put together and professional for new users and provide a way for existing users to feel like they can talk to other users of the app about the app. A terms and conditions is also really important because since it's anonymous it could be really dangerous, therefore, I want to be protected legally and morally from what people use the app to say. In accordance with my first sentence, the documentation would provide new users a way to learn about the app and understand how to use it better incase they are not fully comfortable with the user interface or don't immediately understand how to use the app such as creating an account using the web interface or if something is not working for them.

Conclusion

In conclusion, these improvements which I have outlined above could make a very meaningful difference for new and existing users in the use of the solution. These solutions are not changes to actually applications they are more like features that could be additions to the current application. Updates such as these are very important in a multitude of reasons. It helps make the application more active, and it also attracts new users if they know it will be updated frequently.
